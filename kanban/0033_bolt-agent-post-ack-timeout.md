# Bolt カスタムエージェントのack()後タイムアウト時間

## 知りたいこと

Boltで構築したカスタムエージェントにおいて3秒以内にack()をした後、タイムアウトは何秒なのか

## 目的

非同期で実行させるのがよいのがわかったが、ack()後の残り時間がわかっていないため。

## 調査サマリー

カスタムエージェントが使う `message.im` は Events API のイベントであり、**ack()後にSlack側から課せられる処理タイムアウトは存在しない**。

| タイムアウト | 時間 |
|------------|------|
| ack() 呼び出し期限（HTTP 200応答） | **3秒以内** |
| `setStatus` のローディング表示自動クリア | **2分後（メッセージ未送信時）** |
| Slackが課す処理タイムアウト | **なし** |
| Lambda タイムアウト（インフラ制約） | 最大15分（設定依存） |

スラッシュコマンドやブロックアクション（`response_url` を持つもの）には「30分・最大5回」という制限があるが、`message.im` イベントには適用されない。

`setStatus` のローディング表示は2分後に自動消滅するが、処理そのものは止まらない。2分を超える処理が見込まれる場合はストリーミングや中間結果送信を検討する。

## 完了サマリー

- 調査日: 2026-04-16
- 結論: ack()後のSlack側タイムアウトなし。setStatusのローディング表示は2分で自動クリア。インフラ（Lambda等）のタイムアウトが実質的な制約となる
