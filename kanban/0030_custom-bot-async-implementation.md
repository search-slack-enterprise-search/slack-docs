# カスタムボット非同期実装方法

## 知りたいこと

カスタムボットの非同期での実装方法

## 目的

ボットを非同期で動かす場合、ボット側はイベント受信と分けると回答を返すべき先がわからず、回答があるまでユーザーは何も動いていないように見えると思われる。どうやって実装したらいいかしりたい。またボットは非推奨になるって聞いたけど、Agentしか使えなくなるってこと？

## 調査サマリー

### 非同期実装：イベントペイロードに返答先情報が含まれる

Slackのイベント・インタラクションペイロードには、返答先（チャンネルID・スレッドTS）が最初から含まれているため、「どこに返せばいいかわからない」という問題は発生しない。

- `event['channel']` + `event['ts']`（または `event['thread_ts']`）でチャンネルとスレッドを特定できる
- まず `ack()` で即座に確認応答（3秒以内必須）、その後に非同期で処理し、`chat.postMessage` で返答する

### 処理中の表示方法

| 方法 | 用途 |
|---|---|
| `assistant.threads.setStatus` | Agent コンテナ（Agents & AI Apps有効時）でのTyping Indicator的な表示。返答送信時に自動クリア |
| `SayStream` | テキストを段階的にストリーミング送信（LLM生成のリアルタイム表示等） |
| `chat.postMessage` → `chat.update` | 「処理中...」メッセージを先投稿し、完了後に更新 |

### 遅延レスポンスの主なパターン

1. **response_url / respond()**: Slash Command・Block Action のペイロードに含まれる。30分以内・5回まで使用可能。`respond()` はBoltのラッパー
2. **chat.postMessage 直接呼出**: イベント（app_mention等）や30分超の処理に対応。チャンネルIDとTSはイベントペイロードから取得
3. **trigger_id + Modal**: ユーザーへの確認フローが必要な場合

### ボットの非推奨について

**ドキュメント内に「ボットuser廃止予定」の明示的な記述は見つからなかった。** 現状は:

- 従来のBot user + Events APIによる実装は引き続き利用可能
- 新しいAI機能（Agents & AI Apps）は有料プランが必要な新しいオプションとして追加されている
- RTM APIはレガシーカテゴリーへ移動済み（廃止済み）
- Slackドキュメントでは「Agentがbotを置き換える」という方向性は示されているが、既存botの廃止スケジュールは未記載

## 完了サマリー

- **ログ**: `logs/0030_custom-bot-async-implementation.md`
- **結論**:
  - 非同期実装ではイベントペイロードに返答先（channel + ts）が含まれるため迷う必要はない
  - `ack()` → 非同期処理 → `chat.postMessage` が基本パターン
  - 処理中表示は `assistant.threads.setStatus`（Agent）または `chat.update` パターンで対応
  - ボット非推奨の明示的な廃止アナウンスはドキュメント内に見当たらない。現在はAgentが新しい推奨オプション（有料）として追加されており、既存botは引き続き動作する
