# カスタムエージェントで LLM は必須か（回答を自前で組み立てても良いのか）

## 知りたいこと

0023番への更問い。回答テキストを作成できるのであればLLMを使う必要はないということか。

## 目的

回答を見る限り、必ずしもLLMを使う必要がないように見える。またMCPへの比較を出したのは、MCPはあくまでLLMが回答を作るための情報を提供するまでがMCP Serverの役目であり、カスタムエージェントは回答を作るための情報までで良いのか、(LLM使ったりプログラムで組み立てたりして)回答をすべて作成する必要があるのか、そこを特定したかった。

## 調査サマリー

### 結論

**LLM は技術的に必須ではない。ただしカスタムエージェントは「完全な回答の作成・送信」まで責任を持つ必要がある。**

### Q1: LLM を使わなくても良いか？

**YES（技術的には必須ではない）。**

ドキュメントの記述（`developing-agents.md` 87行目）:
> "choose which other messages from the conversation to include in the **LLM prompt or your app logic**"

「LLM prompt **または** your app logic」と明示されており、プログラムで固定テキストを返すことも選択肢として認められている。`say(text=...)` に渡すテキストはどんな方法で生成しても良い。

ただし概念的区分として：
- LLM なし（プログラム固定応答）= Slack の定義では「**Bot**」
- LLM あり（推論・ツール使用）= Slack の定義では「**Agent**」

### Q2: カスタムエージェントの責任範囲は？

**「回答の完全作成・送信」まで必須。「情報提供まで」ではない。**

| 役割 | MCP Server | カスタムエージェント |
|------|-----------|-------------------|
| 相手 | LLM（AI モデル） | ユーザー（人間） |
| 役割 | データ・ツールの提供 | ユーザーとの対話・**完全な回答の送信** |
| 回答生成責任 | **なし** | **あり（必須）** |

MCP Server は「LLM にデータを渡すまで」が役割。カスタムエージェントはユーザーへの**完全な回答を `say()` で送信するまで**が責任範囲。

### Q3: 「コンテキストを Slack AI に渡して回答を作らせる」構造はあるか？

**ない（ドキュメント上に存在しない）。** カスタムエージェントは自前で回答を生成・送信する必要がある。

### まとめ図

```
【MCP Server の構造】
ユーザー → LLM → MCP Server（データ提供）→ LLM（回答生成）→ ユーザー
            MCP Server はユーザーと直接やり取りしない

【カスタムエージェントの構造】
ユーザー → カスタムエージェント → [任意: LLM / 外部API / プログラム処理]
                               → 完全な回答を生成 → say() → ユーザー
            カスタムエージェントが完全な回答の送信まで担う
```

## 完了サマリー

**調査完了日**: 2026-04-16

カスタムエージェントで LLM は技術的に必須ではない（ドキュメントも「LLM prompt or your app logic」と記述）。プログラムで固定テキストを生成しても動作するが、Slack の概念定義では LLM なしのものを「Bot」と呼ぶ。重要な点は責任範囲の違い：MCP Server は「LLM にデータを渡すまで」が役割でユーザーへの回答生成責任はないが、カスタムエージェントは「完全な回答を `say()` でユーザーに送信するまで」が必須の責任。「コンテキストを Slack AI に渡して回答を作らせる」という構造は現時点ではドキュメント上に存在しない。
